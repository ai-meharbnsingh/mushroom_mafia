## STEP 1: ADMIN FLASHES DEVICE (Dashboard)

### What Happens

Admin connects a **fresh ESP32** via USB to their computer, opens the dashboard in Chrome, and clicks "Flash Device". Everything happens in one session.

### Detailed Flow

```
Admin opens Dashboard → Devices → "Flash New Device"

  1. CONNECT
     │  Browser prompts: "Select USB port"
     │  Admin selects the ESP32's serial port (e.g., /dev/cu.usbserial-0001)
     │  Web Serial API connects at 115200 baud
     │
  2. FLASH FIRMWARE
     │  Dashboard fetches latest .bin from: GET /api/v1/firmware/latest
     │  Uses esptool-js to flash .bin to ESP32 via Web Serial
     │  Progress bar shows flashing progress (0% → 100%)
     │  ~30-60 seconds for 1.5MB firmware
     │
  3. READ MAC ADDRESS
     │  After flash, ESP32 reboots
     │  Dashboard reads Serial output (115200 baud)
     │  Firmware prints: "MAC: AA:BB:CC:DD:EE:FF" on first boot
     │  Dashboard captures MAC address from Serial stream
     │
  4. REGISTER DEVICE IN BACKEND
     │  Dashboard calls: POST /api/v1/devices/provision
     │  Body: { mac_address: "AA:BB:CC:DD:EE:FF", device_type: "ESP32" }
     │  Backend returns: { device_id, license_key: "LIC-877V-4REX-K60T" }
     │
  5. WRITE LICENSE KEY TO ESP32
     │  Dashboard sends via Web Serial: "KEY:LIC-877V-4REX-K60T\n"
     │  Firmware receives it, writes to EEPROM at ADDR_KEY_FLAG
     │  Firmware responds: "KEY_OK:LIC-877V-4REX-K60T"
     │  Dashboard confirms key was written
     │
  6. GENERATE STICKER DATA
     │  WiFi AP Name: "MUSH_" + last 4 chars of key = "MUSH_K60T"
     │  WiFi Password: "123456" (standard for all devices)
     │  QR Code: Contains license key "LIC-877V-4REX-K60T"
     │  Barcode: Code128 encoding of license key
     │
  7. PRINT STICKER
     │  Dashboard opens thermal printer layout
     │  Sends to connected label printer via Browser Print API
     │
  8. ESP32 REBOOTS INTO CAPTIVE PORTAL MODE
     │  Firmware detects: has license key, no WiFi credentials
     │  Starts AP: "MUSH_K60T" with password "123456"
     │  Ready for Step 2
     │
  ✓ DONE — Device flashed, registered, sticker printed
```

### The Chicken-and-Egg Problem (CRITICAL)

**Problem:** License key is generated by backend AFTER flashing, but firmware needs the key.

**Solution:** Two-phase Serial communication.

```
Phase 1: Flash firmware (esptool-js writes .bin to flash memory)
Phase 2: Write config (Dashboard sends license key via Serial command)

The firmware has a "Setup Mode" on first boot:
  - If EEPROM has no license key (addr 13 == 255)
  - Firmware enters Setup Mode
  - Listens on Serial for "KEY:LIC-XXXX-YYYY-ZZZZ\n"
  - Writes key to EEPROM
  - Responds "KEY_OK:LIC-XXXX-YYYY-ZZZZ"
  - Reboots into normal Captive Portal mode
```

### Firmware Changes for Setup Mode

```cpp
// In main.ino setup(), after eepromInit():

if (EEPROM.read(ADDR_KEY_FLAG) == 255) {
    // No license key — enter Setup Mode
    Serial.println("=== SETUP MODE ===");
    Serial.print("MAC:");
    Serial.println(WiFi.macAddress());  // Dashboard reads this

    // Wait for license key from Serial (Web Serial API)
    lcd.clear();
    lcd.print("SETUP MODE");
    lcd.setCursor(0, 1);
    lcd.print("Waiting for key...");

    while (true) {
        if (Serial.available()) {
            String input = Serial.readStringUntil('\n');
            input.trim();
            if (input.startsWith("KEY:")) {
                String key = input.substring(4);
                if (key.length() == 18 && key.startsWith("LIC-")) {
                    // Write to EEPROM
                    char keyArr[20];
                    key.toCharArray(keyArr, 20);
                    writeStringToEEPROM(ADDR_KEY_FLAG, keyArr);
                    EEPROM.commit();

                    Serial.println("KEY_OK:" + key);
                    lcd.clear();
                    lcd.print("Key Saved!");
                    lcd.setCursor(0, 1);
                    lcd.print(key);
                    delay(2000);
                    ESP.restart();  // Reboot into captive portal
                } else {
                    Serial.println("KEY_ERR:Invalid format");
                }
            }
        }
        delay(10);
    }
}
```

### Firmware Changes for WiFi AP Name

```cpp
// In captivePortal.ino → startCaptivePortal():
// BEFORE: WiFi AP name = "MUSHROOM_" + last 5 MAC chars
// AFTER:  WiFi AP name = "MUSH_" + last 4 chars of license key

String apName = "MUSH_";
if (strlen(licenseKey) >= 4) {
    apName += String(licenseKey + strlen(licenseKey) - 4);
} else {
    // Fallback to MAC-based name
    String mac = WiFi.macAddress();
    String suffix = mac.substring(mac.length() - 5);
    suffix.replace(":", "");
    apName += suffix;
}

// BEFORE: Password = "mush_" + last 4 chars of key (variable per device)
// AFTER:  Password = "123456" (standard for all devices)
String apPass = "123456";
```

## STEP 3: DEVICE PINGS BACKEND — "I'M READY" (Automatic)

### What Happens

Once connected to the internet, the device automatically registers with the backend. No user action needed.

### Detailed Flow

```
Device boots with WiFi connected:

  1. REGISTER WITH BACKEND
     │  POST https://protective-enjoyment-...up.railway.app/api/v1/device/register
     │  Headers: X-Device-Key: LIC-877V-4REX-K60T
     │  Body: {
     │    "license_key": "LIC-877V-4REX-K60T",
     │    "mac_address": "AA:BB:CC:DD:EE:FF",
     │    "firmware_version": "4.0.0",
     │    "hardware_version": "ESP32-WROOM-32"
     │  }
     │  Response: { "status": "ok", "device_id": 3 }
     │
  2. BACKEND UPDATES DEVICE RECORD
     │  Sets: is_online = true
     │  Sets: last_seen = now()
     │  Sets: firmware_version = "4.0.0"
     │  Sets: device_ip = (from request or telemetry)
     │  Status remains: PENDING (waiting to be linked to a room)
     │
  3. DEVICE ENTERS HTTP BOOTSTRAP MODE
     │  Sends sensor readings via HTTP POST every 30 seconds
     │  Polls GET /device/provision/{license_key} every 30 seconds
     │  Response: { "status": "pending" } (not yet linked/approved)
     │
  4. DASHBOARD SHOWS DEVICE AS "ONLINE - PENDING"
     │  WebSocket broadcasts: { event: "device_registered", payload: {...} }
     │  Devices page shows: ESP32-Sensor-003 | PENDING | Online
     │  LCD shows: "WiFi: OK | HTTP Mode | Sending Data"
```

## DEPENDENCIES & REQUIREMENTS

### Hardware Required
- Chrome/Edge browser (Web Serial API not in Firefox/Safari)
- USB cable (micro-USB or USB-C depending on ESP32 board)
- Thermal label printer (Brother QL-series or Zebra ZD-series recommended)
- 62mm continuous label roll

### Browser Requirements
- Chrome 89+ or Edge 89+ (for Web Serial API)
- Camera access (for QR scanning)
- HTTPS context (Web Serial requires secure context — works on localhost for dev)

### Security Notes
- Web Serial API requires user gesture (click) — cannot be triggered automatically
- Camera requires explicit user permission
- Standard WiFi password "123456" is intentionally simple — the captive portal is temporary and local only
- After WiFi setup, the AP is shut down — no ongoing security risk
